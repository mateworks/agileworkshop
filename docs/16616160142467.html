Sprint计划会

2022/08/28 posted in  敏捷基础

顾名思义，计划会就是做计划的会，目的是输出冲刺工作计划。

冲刺计划会是冲刺过程中的第一个重要仪式，它的主要目的是确定下个冲刺的目标和冲刺内容，决定着整个团队的努力方向，如何强调其重要性都不为过，一个高质量的计划会可以让团队冲刺工作事半功倍，相反一个糟糕的计划会则可能变成压榨团队的工具。

如何组织计划会

谁来参加

计划会是敏捷团队的全员会，参与冲刺工作的全员都要参加，包括产品负责人、Scrum Master和开发团队，其中开发团队根据产品类型可以包括开发工程师、测试工程师、UE、UI等角色。

如果其他管理人员和客户代表有兴趣也可以参加。

什么时候召开

冲刺计划会需要在冲刺内尽早召开，一般在冲刺第一天的上午开始。Scrum Master需要在会议开始前做好会议准备：会议室、确保会议召开前提满足、通知大家会议时间及议程等等。

会议准备

为了确保计划会高效，团队需要在团队公约内明确定义计划会召开的条件，以便于让各个角色遵守。下面是一些建议的会前需要完成的准备，给大家做参考：

  • PO完成产品待办列表梳理，已完成优先级排序。
  • 按照与团队约定的需求准入条件（DoR）准备。
  • 产品待办列表内高优先级故事量至少足够团队1-2个冲刺的消耗。
  • 团队已经制定了清晰的冲刺完成标准(DoD)
  • Scrum Master需要准备好 团队速率、约束条件、改进计划等信息，已做为团队选择冲刺范围的输入信息。

会议时长

对于一个标准的5-9人的敏捷团队的双周冲刺，计划会一般需要耗时2-4小时，如果团队规模或者冲刺时长更长则可能消耗更多的时间。

当计划会时间超过四小时时，会议效率就会显著下降，效果也会大打折扣，如果团队出现这种情况 Scrum Master需要考虑采取措施提高计划会效率。

会议议程

标准的冲刺计划会包括两部分内容。

第一部分

产品负责人首先和大家同步产品路线图及本冲刺期望达成的目标，然后围绕以上信息就冲刺内容与大家沟通，具体流程如下：

  1. 产品负责人会把待开发的高优先级功能介绍给团队，回答团队提出的问题。
  2. 如果故事尚未估算则Scrum Master组织大家对故事进行估算。
  3. 如果故事过大或过小则需要当场决定如何对故事进行拆分或合并。
  4. 将决定要在下个冲刺开发的故事加入到Sprint Backlog.
  5. 不断充分以上过程，直到团队觉得已经选择了足够冲刺内消耗的故事为止。

第二部分

第二部分一般不需要PO在场。

  1. 团队会在这部分讨论选择的故事的实现思路、内外部依赖、风险等，决定是否能在下一冲刺完成。
  2. 团队根据情况需要判断是否需要加入必须完成的技术故事。
  3. 给每个选择进入冲刺的故事确定唯一的故事负责人（Story Owner)
  4. 如果有必要可以同步完成子任务的拆分与认领。

完成以上过程后，团队需要将最终结果通知PO , 就冲刺目标与冲刺内容达成一致。

会后检查表

  1. 确保所有人都理解冲刺目标级完成标准定义（DoD)
  2. 每个故事已经指定了唯一的Owner（可以在冲刺过程调整）
  3. 确保PO、团队、核心干系人对冲刺目标、内容、完成标准理解一致。
  4. 输出了完整的Sprint Backlog

注意事项

  • 在Scrum开始时候尽早召开计划会
  • 准备进入冲刺的故事必须满足DoR要求，如果不满足，团队可以拒绝。
  • 一般进入计划会的故事应该已经完成了故事点估算，对于未估算或者需要调整点数大小的可以在计划会上进行估算。
  • 每个Sprint的目标应该包括发布可以工作的、经过测试的代码，这些功能可以实现对客户有价值的一些功能。
  • 产品待办列表是PO所有的，但是团队其他人同样可以往待办列表增加故事或者调整优先级，但须需要告知PO并达成一致。
  • 一旦冲刺开启后，只有团队自己能决定是否向Sprint内增加工作，团队外的人无权给团队增加工作。
