迭代评审会

2022/08/28 posted in  敏捷基础

在传统软件开发流程下通常会在关键的里程碑节点被要求做系统演示，进行演示对团队来说会是一股强大的推动力，团队会通过各种方法来确保演示过程的顺利。对敏捷团队来说，定期进行功能进行演示也能带来类似的效果，这可以有效激励团队按时完成迭代计划。Scrum中对应的仪式叫做“迭代评审会” 或 “迭代演示会”，团队会在这个会上向客户、用户、核心干系人展示冲刺成果，听取他们的意见，逐步建立敏捷团队与客户信任关系。

冲刺评审会一般在冲刺结束时候召开，在会议期间，团队会根据用户故事中定义的验收标准（AC）和团队一起评审交付的故事，PO可以选择接受或拒绝已完成的用户故事。同时提出自己的意见与建议。ScrumMaster 会记录大家的意见以作为规划后续冲刺的参考。如何组织冲刺评审会呢？

准备工作

成熟的敏捷团队一般不需要专门花精力进行准备，但是新的敏捷团队一般需要进行一些准备工作以确保演示的顺利，一般SM会在迭代最后一天的站会上会提醒团队进行准备，准备内容包括：

  1. 确定哪些故事需要在评审会上进行展示。
  2. 决定故事的呈现顺序及演示逻辑。
  3. 确定演示负责人，鼓励团队全员上场，或者轮流上场。
  4. 如果有高级管理人员参加则可以提前组织进行一次预演，确保演示过程顺利。

如果开始时候觉得展示所有故事很难，可以选择几个重要的故事进行展示，鼓励团队去思考如何向干系人进行演示，如何说明这些功能的价值，一起完成准备工作，避免准备工作变成某一两个人的事儿。

谁来参加

首先Scrum团队所有人都要参加评审会，这是团队为数不多的对外展示自己工作成果、听取建议的机会，一些团队会觉得这样会浪费团队时间，但是作为敏捷教练要注意维护团队演示自己工作成果的权利，否则评审会就无法起到激励团队的作用。在合适的交付节点可以尝试邀请客户、高级经理参加演示会议，，因为这是团队炫耀他们迭代成绩的好机会，ScrumMaster应该善用这个机会。

引导流程

评审会一般在冲刺最后一天召开，对于一个标准的5-9人的敏捷团队，一次评审会大概1-2小时时间，基本的会议过程如下：

对于新团队或者参与评审会的人员中有较多新面孔的时候，Scrum Master需要在会议开始的时候先简单介绍一下团队目前采用的工作模式，让大家理解目前团队在遵循的迭代流程，让他们理解接下来看到的只是本迭代完成的部分功能，并非产品的全部功能。

  1. 会议开始后，由会议引导人或PO介绍本次迭代的目标、用户故事完成情况，如有需要还需要介绍本冲刺完成的非功能性需求情况，如性能优化、压测数据等。。
  2. 演示负责人（鼓励每个成员都上场进行演示）按照计划的顺序开始演示，边演示边进行说明，这个过程需要注意避免仅零碎的演示系统哪里有变化、哪里新加了功能，要把这些变化和功能代入到业务场景内，重点描述通过这些功能解决了哪些问题。
  3. 演示过程Scrum Master记录PO和其它干系人提出的问题和建议，并在会议结束后同步给参会人员。
  4. 演示顺利结束后可以组织庆祝团队取得的成绩，如果演示过程不顺利则可以在回顾会中讨论存在的问题及改进方法。

常见问题

演示时软件无法正常工作

这是演示会上最常见的问题，非常符合墨菲定律，明明评审会前测试都挺好，一到演示会上就出现各种莫名奇妙的问题，比如网络不通了、数据错误、版本未更新等问题。通常这种情况反应的是两个问题：

  1. 自动化基础设施尚不完善，尚依赖于手工进行集成、测试、部署，这种情况下需要推动团队加强相关自动化能力建设。
  2. 演示会前未对演示所需的必要条件进行检查或有遗漏，这种情况需要规范团队每次演示前的准备工作清单，避免遗漏。

故事无一完成

新的敏捷团队偶尔会出现，冲刺结束了，但是没有一个故事是完全完成了的，这时很多团队会考虑取消这次评审会，一般我们会建议除非完成度特别低，完全没有东西可以演示才考虑取消，否则应该坚持组织演示，即使还有一些故事有遗留缺陷，UI还未完善，流程还没跑通。如果轻易取消演示会让团队产生一个错觉：每个迭代结束都要交付可工作的软件这个承诺并不那么重要。因此出现这种情况时我们应该鼓励团队坦然面对这种情况，并在回顾会上对造成这种情况的原因进行讨论，避免同样的情况再次出现。

演示依赖于其他团队的产品或服务

这种情况通常当系统属于某一个大型产品的一部分的情况，这种情况可以考虑与对方团队一起进行联合演示，这样可以帮助人们从整体视角审视这个产品。还有一种情况是系统依赖于某个外部系统的服务，但是由于排期等各种原因尚未完成对接与联调，这种情况下可以考虑通过Mock方式模拟外部依赖，完成系统的演示。

软件没有界面可用于演示

造成这种情况的原因通常有两种：

  1. 团队故事切分不合理，比如按前后端去拆分故事了，本次只完成了后端的故事，所以无法单独演示，这种情况下需要引导团队改进故事拆分方式，尽量按照切蛋糕的方式，纵向拆分故事。
  2. 需求本身设计就不具有界面，如命令行应用、API、算法等故事，这种情况下确实比较难让客户对交付内容有直观的感受，客户看不懂也就失去了演示的意义，这种情况下建议可以结合这些故事的完成标准、测试数据、运行示例、潜在应用场景等向客户说明故事的内容及价值。
